## Sustained-note isolation: Adaptive pitch region refinement

# PROBLEM:
# When using a frequency-band search to find a sustained pitch target
# (e.g., @findStableRegion scanning for C5 at 450-600 Hz), the found
# region may extend through a pitch transition into an adjacent note
# that also falls within the search band (e.g., B4 at 494 Hz is inside
# a 450-600 Hz band meant for C5 at 523 Hz). This contaminates the
# target measurement by pulling the mean F0 toward the transition.
#
# This problem occurs whenever adjacent melodic pitches are close enough
# to share a frequency band, which is common in stepwise melodies.

## Solution: Two-pass adaptive refinement

# Apply as a post-hoc step after a broad-band region search returns a
# candidate region. The method is general but most needed when adjacent
# pitches are within the search band.
#
# PASS 1 — Forward convergence (establish center pitch):
#   Start from region start. Grow the analysis window one frame at a time.
#   At each step, compute running mean F0 and SD across all frames so far.
#
#   Convergence criterion: Mean F0 has changed less than X% (start with 1%)
#   over the last 10 frames added.
#
#   Output: converged center F0, converged SD, convergence frame count.
#
#   Rationale for growing window: A fixed short window (50-100 ms) is
#   vulnerable to vibrato phase — a 5-7 Hz vibrato cycle is 140-200 ms,
#   so windows shorter than ~300 ms capture less than two full cycles and
#   the mean varies with phase. Growing until convergence inherently
#   adapts to the vibrato rate present in the signal.
#
#   Expected convergence: At typical frame rates (~15 ms/frame with pitch
#   floor 50 Hz), 2-3 vibrato cycles require 19-40 frames (280-600 ms).
#
# PASS 2 — Backward seek (find true region end):
#   Start from the broad region end. Walk backward one frame at a time.
#   At each position, compute local mean F0 using a fixed 10-frame
#   sliding window (the same width used for the convergence check).
#
#   Match criterion: Local mean F0 falls within 1% of the converged
#   mean from Pass 1. Mean-only — do NOT include SD in the gate.
#
#   Output: The first matching frame (walking backward) becomes the
#   refined region end.
#
#   Rationale: B4 at 494 Hz is ~6% below C5 at 523 Hz. A 1% match
#   criterion on mean F0 cleanly excludes it.
#
#   Why mean-only (no SD check): The converged SD from Pass 1 is
#   computed over ~25 frames (~375 ms, ~2 vibrato cycles). A 10-frame
#   backward window captures less than one vibrato cycle, so its local
#   SD varies with vibrato phase. The 1% SD tolerance is tighter than
#   the phase-dependent fluctuation, causing the match to fail even in
#   clean sustained-pitch territory. The mean is phase-insensitive at
#   10-frame resolution because partial-cycle means still average close
#   to the center frequency when the signal is stationary.
#
#   Why 10-frame window (not convergence length): Using the full
#   convergence window (~25 frames) as the backward window creates a
#   ~400 ms penalty — any position where the window overlaps the pitch
#   transition fails, pushing the refined end back by roughly the full
#   window length from the actual departure point. The 10-frame window
#   (~150 ms) localizes the departure to within one vibrato cycle.

## Edge cases

# Region too short for backward pass:
#   If the total region length minus the forward convergence window is
#   less than the convergence window itself, log a warning and accept
#   the broad region unmodified. Flag for manual inspection.
#
# Forward pass never converges:
#   If the forward pass reaches the region end without meeting the
#   convergence criterion, log a warning and accept the broad region.
#
# In both cases, the data goes through but is flagged in the log.

## Procedure signature (reference design)

# procedure refineRegionEnd: .pitchId, .regionStart, .regionEnd,
#     ... .convergencePct, .matchPct
#
# Returns:
#   .refinedEnd (time)
#   .centerF0 (converged mean)
#   .centerSD (converged SD — informational, not used in Pass 2 gate)
#   .converged (0/1)
#   .trimmed (0/1)
#
# Typical values: .convergencePct = 1.0, .matchPct = 1.0
# Note: .matchPct applies to mean F0 only in backward pass.
# Backward window is fixed at 10 frames (not convergence length).

## When to use this pattern

# - Isolating sustained pitch targets from melodic passages
# - Any frequency-band search where adjacent musical intervals
#   may fall within the same band (intervals < minor third apart)
# - Not needed when adjacent pitches are well-separated from the
#   search band (e.g., octave leaps)
#
# Provenance: Designed for A2 batch analyzer "Somewhere" excerpt
# analysis (C5 target adjacent to B4), 25 Feb 2026.
# Validated: v2.8 (convergence-length window + SD gate) was too
# aggressive — trimmed ~400 ms of clean C5. Revised to 10-frame
# window with mean-only gate in v2.9, 25 Feb 2026.

# PROBLEM:
# When isolating a sustained pitch target from a melodic passage,
# the region boundary may fall within a gradual pitch transition
# (e.g., half-step glide from C5 to B4). Detecting the exact
# departure point is complicated by singing vibrato, which creates
# oscillations in the F0 contour that mask the directional pitch
# movement.
#
# This method detects pitch departures as the point of maximum
# negative (or positive) regression slope in the smoothed F0
# contour. It is a single-pass alternative to the two-pass
# adaptive refinement method documented above.

## Method: Gaussian smooth + sliding-window regression slope

# STEP 1 — Smooth the Pitch object:
#   Apply Praat's built-in Gaussian smooth to remove vibrato and
#   frame-to-frame jitter before analysis.
#
#   selectObject: pitchId
#   smoothedPitchId = Smooth: bandwidth
#
#   Bandwidth parameter (Hz): Sets the cutoff frequency of the
#   Gaussian filter. Energy above this frequency is attenuated.
#   Choose bandwidth < lowest expected vibrato rate.
#
#   Canonical: bandwidth = 2 Hz
#   Rationale: Singing vibrato ranges 4.5–8 Hz. A 2 Hz cutoff
#   sits well below the lowest vibrato rate, suppressing all
#   vibrato uniformly regardless of rate. Also suppresses
#   noise-induced F0 jitter (which is broadband above 0 Hz).
#
#   Suppression performance (benchmarked 26 Feb 2026):
#     ±0.5st @ 6Hz vibrato: 550x suppression, post-smooth SD < 0.001 st
#     ±1.0st @ 4.5Hz vibrato: 150x suppression, post-smooth SD < 0.005 st
#     Additive noise (HNR ~30dB): no measurable impact on detection
#
# STEP 2 — Extract smoothed F0 in semitones:
#   selectObject: smoothedPitchId
#   f0St# = List values in all frames: "semitones re 440 Hz"
#
# STEP 3 — Compute sliding-window regression slope:
#   For each frame i, fit a linear regression over frames
#   [i - halfWin, i + halfWin] and record the slope (st/frame).
#
#   Canonical: slopeWindow = 10 frames (halfWin = 5)
#
#   The slope at frame i is:
#     slope[i] = sum((j - i) * f0St[j]) / sum((j - i)^2)
#   for j from (i - halfWin) to (i + halfWin).
#
# STEP 4 — Find slope minimum (departure) or maximum (arrival):
#   Search within the expected region for the frame with the most
#   negative slope (pitch falling) or most positive slope (pitch
#   rising). This is the detected departure/arrival point.
#
# STEP 5 — Latency bias correction:
#   The centered regression window introduces a systematic delay
#   of halfWin frames. Correct by subtracting:
#     correctedTime = detectedTime - (halfWin * timeStep)
#
#   At canonical settings (halfWin=5, timeStep~0.015s): ~45ms bias.
#   After correction, detection error approaches zero on synthetic
#   benchmarks.

## Confidence verification (recommended for production use)

# Three checks, applied per detection:
#
# A) SNR check:
#    Measure the noise floor as max(|slope|) during a known-stable
#    region (sustained pitch, no transitions). SNR = |departureSlope|
#    / noiseFloor. Thresholds:
#      HIGH: SNR >= 10x
#      MED:  SNR >= 3x
#      LOW:  SNR < 3x (unreliable — flag for manual inspection)
#
# B) Suppression check:
#    Compute SD of the F0 contour in the stable region before and
#    after smoothing. suppressionRatio = rawSD / smoothedSD.
#    Expect ratio > 100x for typical vibrato. Ratio < 10x suggests
#    smoothing is ineffective (bandwidth too high, or no vibrato
#    to suppress).
#
# C) Spatial check:
#    Verify that the detected time falls within the expected
#    transition zone (±100ms of the expected departure region).
#    Detections outside this zone may indicate false positives
#    from onset/offset artifacts.
#
# Overall: HIGH requires SNR >= 10x AND spatial pass.

## Comparison with two-pass adaptive refinement

# Two-pass method (documented above):
#   + No vibrato-rate knowledge needed
#   + Self-calibrating via forward convergence
#   + Validated on real Somewhere data
#   - Two passes, more computation
#   - Convergence window can be long (~25 frames)
#
# Smooth + slope method:
#   + Single pass, simpler implementation
#   + One Praat command replaces custom smoothing procedures
#   + Handles all vibrato rates uniformly (no window tuning)
#   + Robust to additive noise
#   + Built-in confidence verification
#   - Requires bandwidth parameter choice (but 2 Hz is robust default)
#   - Systematic latency bias (correctable)
#   - Not yet validated on real voice data (synthetic benchmark only)
#
# Recommendation: Use Smooth + slope as the primary method. Fall back
# to two-pass convergence if confidence is LOW, or when the expected
# transition location is unknown (two-pass is self-localizing; slope
# method requires a search region).
#
# Provenance: Developed via synthetic benchmark (C5 -> B4 half-step,
# 6 vibrato configurations, 0 to ±1.0st at 4.5-8Hz), 26 Feb 2026.
# Gaussian smooth at 2 Hz achieved 6/6 HIGH confidence with and
# without additive noise. Median filter and moving average were
# tested and found inferior (median: window-size dependent, noise-
# permeable; moving average: sinc sidelobes leak vibrato residual).
