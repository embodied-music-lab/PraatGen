## 1. PITCH EXTRACTION

# ALGORITHM SELECTION GUIDE
#
# Which pitch algorithm to use depends on the PURPOSE of the analysis.
# Source: Praat manual, "how to choose a pitch analysis method"
# (https://www.fon.hum.uva.nl/praat/manual/how_to_choose_a_pitch_analysis_method.html)
#
#   PURPOSE                              ALGORITHM                    SECTION
#   -----------------------------------  ---------------------------  -------
#   F0 tracking, intonation, prosody,    Filtered autocorrelation     §1A
#     melodic analysis
#
#   Voice quality input (jitter,         Raw cross-correlation        §1B
#     shimmer, HNR via PointProcess)
#     — uses shorter windows for
#       precise period alignment
#
#   Raw periodicity (non-speech          Raw autocorrelation          §1C
#     signals, missing fundamentals)
#
#   Filtered cross-correlation           Not recommended              —
#     — no known preferred use case
#     (Praat manual: experimental)
#
# If the task requires BOTH F0 tracking AND voice quality measures,
# use TWO SEPARATE pitch objects — one per algorithm. Never reuse a
# filtered autocorrelation Pitch for jitter/shimmer input, or a raw
# cross-correlation Pitch for intonation analysis.
#
# If the purpose is ambiguous, ASK THE USER before selecting.

### 1A. Pitch (filtered autocorrelation) — for F0 contour tracking, intonation

# Use case: F0 tracking, prosody, intonation contours
# NOTE: Parameter is "pitch top" (NOT "pitch ceiling")
# NOTE: Requires 11 parameters — "voiced unvoiced cost" is the 11th

To Pitch (filtered autocorrelation): 0.0, 50, 800, 15, "no", 0.03, 0.09, 0.5, 0.055, 0.35, 0.14

# Parameter breakdown (Praat GUI defaults, verified 25 Feb 2026):
#   time step = 0.0 (auto: 0.75 / pitch floor)
#   pitch floor = 50 Hz
#   pitch top = 800 Hz            ← NOT "pitch ceiling"
#   max candidates = 15
#   very accurate = "no"
#   attenuation at top = 0.03
#   silence threshold = 0.09
#   voicing threshold = 0.5
#   octave cost = 0.055
#   octave jump cost = 0.35
#   voiced unvoiced cost = 0.14
#
# CRITICAL — 2× PITCH TOP RULE (FAC only):
# The filtered autocorrelation algorithm applies an internal low-pass
# filter at pitch top. Attenuation begins at pitch_top / 2. This means:
#   - pitch top = 800 → reliable tracking degrades above ~400 Hz
#   - To track F0 at frequency X, set pitch top ≥ 2 × X
#
# Examples:
#   Speech only (typical adult):     pitch floor = 75, pitch top = 600
#   Higher voices (child, soprano):  pitch floor = 50, pitch top = 800
#   Singing up to C5 (523 Hz):      pitch top ≥ 1050 Hz
#   Singing up to C6 (1047 Hz):     pitch top ≥ 2100 Hz
#   Lower voices (bass):            pitch floor = 50, pitch top = 400
#
# The 2× rule applies ONLY to filtered autocorrelation because of its
# internal LPF. Raw cross-correlation and raw autocorrelation use a hard
# cutoff at pitch ceiling — see §1B and §1C.
#
# THESE PARAMETERS DIFFER SUBSTANTIALLY from raw cross-correlation and
# raw autocorrelation defaults. Do NOT copy parameters between algorithms.
#
# Always solicit the expected pitch range from the user before selecting
# pitch floor and pitch top values.

### 1B. Pitch (raw cross-correlation) — for voice quality input (jitter/shimmer)

# Use case: Input to PointProcess for jitter, shimmer, voice quality measures
# NOTE: Parameter is "pitch ceiling" (NOT "pitch top")
# NOTE: Requires 10 parameters (no "attenuation at top")

To Pitch (raw cross-correlation): 0.0, 75, 600, 15, "no", 0.03, 0.45, 0.01, 0.35, 0.14

# Parameter breakdown (Praat GUI defaults, verified 25 Feb 2026):
#   time step = 0.0 (auto)
#   pitch floor = 75 Hz
#   pitch ceiling = 600 Hz        ← NOT "pitch top"; hard cutoff
#   max candidates = 15
#   very accurate = "no"
#   silence threshold = 0.03
#   voicing threshold = 0.45
#   octave cost = 0.01
#   octave jump cost = 0.35
#   voiced unvoiced cost = 0.14
#
# Rationale: raw cross-correlation avoids the spectral weighting that
# filtered autocorrelation applies, giving more accurate period detection
# for perturbation measures.
#
# Pitch ceiling is a HARD CUTOFF — no LPF attenuation. Set it to exceed
# the highest expected F0. For singing tasks, raise pitch ceiling to
# cover the vocal range (e.g., soprano to C6 → pitch ceiling = 1100 Hz).
#
# Always solicit the expected pitch range from the user.

## 4. FORMANT ANALYSIS

# ALGORITHM ROUTING (hard):
#
# Choose the formant analysis algorithm based on whether the formant
# ceiling is known:
#
#   CONDITION                             ALGORITHM            SECTION
#   ------------------------------------  -------------------  -------
#   Ceiling uncertain, gender unknown,    FormantPath (burg)   §4A
#     no protocol requirement                                  [DEFAULT]
#
#   Specific ceiling required             Formant (burg)       §4B
#     (replication, protocol, user
#     request)
#
# DEFAULT: FormantPath (burg). This algorithm searches across a range
# of formant ceilings and selects the optimal one per frame. No user
# decision about vocal tract size or gender is needed. This is the
# correct default for any script where the formant ceiling is not
# predetermined by a specific protocol.
#
# OVERRIDE: Formant (burg) with explicit ceiling. Use only when:
#   1. Replicating a published study that used a specific ceiling
#   2. Protocol requires identical ceiling across all participants
#      for between-subject comparability
#   3. User explicitly requests a fixed ceiling
#
# RATIONALE: The previous version of §4 mandated manual ceiling
# selection ("Smaller / Larger" optionmenu) as the universal pattern.
# This forced a Tier 3 decision (estimate a physiological parameter)
# into every formant analysis script, even when the user had no basis
# for making the choice. FormantPath eliminates this friction by
# automating the ceiling search — which is exactly the problem it was
# designed to solve. Per Appendix F's progressive disclosure philosophy,
# parameters that an algorithm can optimize should not be presented to
# the user as mandatory choices.

### 4A. FormantPath (burg) — DEFAULT (automated ceiling optimization)

# Use case: Any formant analysis where the ceiling is not protocol-specified.
# The algorithm evaluates multiple ceilings and selects the optimal one
# per analysis frame based on formant track smoothness.

To FormantPath (burg): 0.005, 5.0, 5500, 0.025, 50, 0.05, 4

# Parameter breakdown (Praat GUI defaults, verified 21 Mar 2026):
#   time step = 0.005 s
#   max number of formants = 5.0
#   middle formant ceiling = 5500 Hz    (center of ceiling search range)
#   window length = 0.025 s
#   pre-emphasis from = 50 Hz
#   ceiling step size = 0.05            (exponential step between candidates)
#   number of steps up/down = 4         (4 steps above and below middle)
#
# CEILING SEARCH RANGE:
#   The algorithm evaluates 2 * nSteps + 1 = 9 candidate ceilings:
#     min ceiling = 5500 * exp(-0.05 * 4) ≈ 4510 Hz
#     max ceiling = 5500 * exp(+0.05 * 4) ≈ 6722 Hz
#
#   This range covers the typical adult male vocal tract (5000 Hz)
#   through adult female (5500 Hz) and somewhat beyond. For children
#   or very small vocal tracts, increase middle ceiling to 6500–8000 Hz
#   or increase nSteps.
#
# QUERY WORKFLOW:
#   After creating the FormantPath, extract the Formant object and
#   query it using standard Formant commands:
#
#     selectObject: soundId
#     fpId = noprogress To FormantPath (burg): 0.005, 5.0, 5500,
#         ... 0.025, 50, 0.05, 4
#     selectObject: fpId
#     formantId = Extract Formant
#     selectObject: formantId
#     meanF1 = Get mean: 1, 0, 0, "Hertz"
#     meanF2 = Get mean: 2, 0, 0, "Hertz"
#
# ⚠️ BUG (Praat 6.4.60–6.4.62):
#   Extract Formant segfaults after Set path: or Set optimal path:.
#   WORKAROUND: Call Extract Formant IMMEDIATELY after FormantPath
#   creation, before any Set command. See COMMANDS_Formant.txt for
#   details.
#
# OPTIMAL CEILING QUERY (optional, for reporting):
#   To report which ceiling was selected:
#     selectObject: fpId
#     optCeiling = Get optimal ceiling: 0.0, 0.0, "3 3 3", 1.25
#     appendInfoLine: "Optimal formant ceiling: ",
#         ... fixed$ (optCeiling, 0), " Hz"

### 4D. FormantModeler — formant tracking quality assessment

# FormantModeler fits polynomial curves to formant tracks and provides
# goodness-of-fit metrics: R-squared, residual standard deviation,
# outlier detection, track separation. These quantify how reliable the
# formant tracking is — a crucial confidence indicator when reporting
# formant means.
#
# CREATION:
#   selectObject: formantId
#   fmId = To FormantModeler: startTime, endTime, 3, 3
#   # 3 formants, 3rd-order polynomial
#
# KEY METRICS:
#   R-squared (Get coefficient of determination):
#     High (>0.8) = stable, trustworthy tracking
#     Low (<0.5) = unstable — either tracking errors or the polynomial
#     model is wrong for this data (see limitation below)
#
#   Track standard deviation (Get formant standard deviation):
#     Scatter of raw data around the polynomial, in Hz.
#     Large SD = noisy tracking or formant movement.
#
#   Track separation (Get average distance between tracks):
#     Average Hz between adjacent formant tracks. Low separation
#     suggests tracks are merging — a signature of unreliable tracking.
#
#   Invalid data points (Get number of invalid data points):
#     Frames flagged as outliers (>3 SD from polynomial). High counts
#     indicate tracking problems OR a wrong model (see below).
#
# SCOPE LIMITATION (hard):
#
# FormantModeler assumes formants vary smoothly within the analysis
# window. It is VALID for:
#   - Sustained vowels
#   - Single extracted vowel tokens
#   - Sustained singing tones
#
# It is INVALID for:
#   - Connected speech spanning multiple vowels
#   - Vowel sequences with large transitions (e.g., /u i u/)
#   - Any signal where formants change rapidly by design
#
# On connected speech, the polynomial smooths away real formant
# transitions and flags real vowel targets as outliers. Low R-squared
# in this context does NOT mean tracking failed — it means the model
# is wrong for the data. If FormantModeler is needed on connected
# speech, segment by vowel first and apply per segment.
#
# REPORTING:
# When FormantModeler metrics are included in script output, report
# them alongside the formant means as confidence indicators:
#
#   appendInfoLine: "F1 mean: ", fixed$ (meanF1, 0), " Hz",
#       ... " (R-sq = ", fixed$ (rSqF1, 2), ")"
#
# Provenance: Empirical testing on sustained vowels (R-sq > 0.9) and
# connected speech /u i u/ sequence (R-sq < 0.3, all transitions
# flagged as outliers). 21 April 2026.

# ----------------------------------------------------------------------------
# 10A. INPUT REQUIREMENTS
# ----------------------------------------------------------------------------
#
# AVQI requires TWO recordings per speaker:
#   1. Sustained vowel (named "sv") — /a/ phonation
#   2. Continuous speech (named "cs") — connected speech sample
#
# The script concatenates voiced portions of both into a single analysis
# signal ("avqi") before computing acoustic measures.
#
# Sustained vowel handling:
#   - If duration > 3 seconds, extracts the LAST 3 seconds
#   - If duration <= 3 seconds, uses the entire recording
#
# Continuous speech handling:
#   - Silence detection: To TextGrid (silences): 50, 0.003, -25, 0.1, 0.1,
#     "silence", "sounding"
#   - Voiced segment extraction via power thresholding (30% of global power)
#     and zero-crossing rate filtering (< 3000 crossings/s)
#   - 30 ms sliding window for voiced/unvoiced classification
#   - Voiced segments concatenated into a single signal

# ----------------------------------------------------------------------------
# 10B. HIGH-PASS FILTER (applied to BOTH recordings before analysis)
# ----------------------------------------------------------------------------

# Filter (stop Hann band): 0, 34, 0.1
#
#   from frequency = 0 Hz
#   to frequency = 34 Hz
#   smoothing = 0.1 Hz
#
# This removes subsonic noise and DC offset. Applied to BOTH cs and sv
# BEFORE any analysis. This filter is NOT used in standard clinical
# analysis (Appendix D §§1–6) — it is AVQI-specific.

# avqi = ((3.295 - (0.111 * cpps) - (0.073 * hnr) - (0.213 * shim)

#   cpps  = Smoothed Cepstral Peak Prominence (dB) — Section 10C

#   shdb  = Shimmer local dB (dB) — Section 10F
#   slope = LTAS slope (dB) — Section 10D
#   tilt  = LTAS trendline tilt (dB) — Section 10E
