# RESERVED IDENTIFIERS (not functions — language constants)
# The following cannot be used as variable names, procedure parameters,
# or loop counters:
#   e         — Euler's number (2.71828...)
#   pi        — Pi (3.14159...)
#   undefined — The undefined value
# Error: "You cannot use 'e' as the name of a variable."
# Common collision: loop counters (for e from 1 to n), procedure
# parameters (.e), single-letter nested loop iterators.
# Provenance: Empirical testing, 3 April 2026.
#
# ============================================================================
# CATALOGUE GHOSTS (verified non-functional in Praat 6.4.65)
# ============================================================================
# These names appear in the Praat source code's function tables and/or in
# the official Functions manual page, but throw "Unknown function" or
# "Symbol without action" when invoked at runtime. Empirically tested in
# Praat 6.4.65 sandbox on 17 May 2026 across 75+ function probes.
# DO NOT USE. Workarounds noted where they exist.
#
#   inner## (a##, b##) — "Unknown function". For matrix product use
#     mul## (a##, b##). For matrix inner product use
#     sum (a## * b##) (elementwise * then sum).
#
#   object# (id), object## (id), object$# (id) — "Unknown function".
#     Scalar object (id) and object$ (id) DO work for cell access.
#     For bulk extraction, use class-specific Get commands in a loop or
#     To Matrix / To TableOfReal to convert object → matrix variable.
#
#   linear## (...) — "Symbol without action" (parser recognizes the name
#     but no implementation is registered). Manual page documents it as
#     "matrix filled by a supplier function" but the supplier syntax
#     cannot be expressed. Use zero## (nrow, ncol) then fill via a
#     for-loop, or use a Formula: ~expression on a Matrix object.
#
#   linear$# (a#, precision), linear$## (a##, precision) — "Unknown
#     function". Use fixed$ (a#, precision) or fixed$ (a##, precision);
#     fixed$ is the actual implementation and accepts both scalar and
#     vector/matrix arguments.
#
#   number# (a$#) — "Unknown function". The official manual page
#     documents this name, but the actual implementation is numbers#
#     (see SECTION 1). Use numbers# (a$#) instead.
#
#   randomPoisson# (n, mean), randomPoisson## (nrow, ncol, mean) —
#     "Unknown function". Both forms are documented on the official
#     manual page but neither exists at runtime in 6.4.65. Only the
#     scalar randomPoisson (mean) works. Workaround: scalar form in
#     a for-loop populating a zero# / zero## vector/matrix.
#
#   repeat## (m##, n) — "Unknown function". Only repeat# (v#, n) is
#     implemented. For matrix tiling, use mul## or construct via
#     element loops over zero## buffer.
#
#   softmax## (m##) — "Unknown function". Use softmaxPerRow## (m##),
#     which is the actual matrix-valued softmax implementation
#     (normalizes each row independently).
#
#   rowInner# (a##, b##) [singular] — "Unknown function". Use the
#     plural rowInners# (a##, b##).
#
#   between# (low, high, n) — "Unknown function". The official names
#     are between_by# (low, high, step) for step-spaced and
#     between_count# (low, high, n) for n-spaced.
#
#   center# (v#) — "Unknown function". The scalar center (v#) returns
#     the center of gravity as a scalar; there is no vector-returning
#     center variant.
#
#   demoExtraControlKeyPressed ( ) — "Unknown function". Use the
#     documented variants: demoCommandKeyPressed, demoOptionKeyPressed,
#     demoShiftKeyPressed.
#
# Cross-reference: catalogue ghosts are flagged in SECTION 1 entries
# with the marker "[CATALOGUE GHOST]" so a reader scanning Section 1
# sees the warning inline with the entry.
#

# Cross-reference companion to Section 1's alphabetical entries.
# Section 1 documents what each function does in isolation.

# native C-speed equivalents.

# Provenance: EML LMM vectorization audit, 15 May 2026, Praat 6.4.65
# sandbox (Linux x86_64). Speedup factors are at typical LMM scale
# (N=600 observations, q=30 random-effect levels, p=5 fixed effects).
# Smaller problems show smaller speedups, but the qualitative ordering
# of patterns is consistent.
#
# Conceptual orientation (vector vs. matrix variables vs. Matrix objects)
# is in Master Prompt Rule 5C. This section assumes that orientation.

# ----------------------------------------------------------------------------
# §4.1  SLICE AND EXTRACT — IDIOMS AND NEGATIVE RESULTS
# ----------------------------------------------------------------------------
#
# Praat has NO slice syntax. Both of the following fail:
#     bad# = M## [1, *]      ✗ "Symbol misplaced"
#     bad# = M## [*, 1]      ✗ "Symbol misplaced"
#
# Use these patterns instead:
#
# Column j of M (n×p) as a vector:
#     col# = rowSums# (part## (M##, 1, n, j, j))
# Empirical speedup vs interpreted loop: 68× at N=600.
#
# Row i of M (n×p) as a vector:
#     row# = columnSums# (part## (M##, i, i, 1, p))
# Empirical speedup vs interpreted loop: 16× at N=600.
#
# Submatrix (rows r1..r2 inclusive, cols c1..c2 inclusive):
#     sub## = part## (M##, r1, r2, c1, c2)
# Native C-speed. Returns a fresh matrix.
#
# Vector subrange (elements i1..i2 inclusive):
#     sub# = part# (v#, i1, i2)
# Native C-speed. Returns a fresh vector.
#
# Note: there is no native column assignment to a matrix. To overwrite
# column j with vector v#, loop:
#     for .i from 1 to n
#         M## [.i, j] = v# [.i]
#     endfor
# Or build the new matrix from parts via mul## / concatenation.

# ----------------------------------------------------------------------------
# §4.2  MULTIPLICATION AND TRANSPOSE
# ----------------------------------------------------------------------------
#
# CAUTION: the * operator between two matrices is ELEMENTWISE, not matrix
# multiplication. Use mul## (A##, B##) for the standard matrix product.
#
#     C## = A## * B##             # elementwise (Hadamard product)
#     C## = mul## (A##, B##)      # matrix multiplication (A·B)
#
# mul_fast## (A##, B##) is ~20% faster than mul## on the same inputs;
# numerical results are identical to working precision. mul_metal##
# (macOS only) uses Apple's Metal Performance Shaders.
#
# Cross-product idiom (X'X):
#     XtX## = mul## (transpose## (X##), X##)
# Empirical speedup vs interpreted loop: 2419× at N=600, p=5.
#
# SigmaG = ZZ' for random-effects covariance:
#     SigmaG## = mul## (Z##, transpose## (Z##))
# Replaces a triple-nested N²×nLev loop. Empirical speedup: 1995× at
# N=600, q=30.
#
# Quadratic form v'Av (returns scalar):
#     val = inner (v#, mul# (A##, v#))
# Two-step: A·v as vector, then inner product with v. Faster than
# expanding the full matrix product.

# ----------------------------------------------------------------------------
# §4.3  SOLVE AND INVERSE
# ----------------------------------------------------------------------------
#
# Linear solve A·x = y (vector RHS):
#     x# = solve# (A##, y#)
#
# Linear solve A·X = Y (matrix RHS):
#     X## = solve## (A##, Y##)
#
# Matrix inverse — DO NOT compute explicitly. Use solve with identity:
#     I## = zero## (n, n)
#     for .i from 1 to n
#         I## [.i, .i] = 1
#     endfor
#     invA## = solve## (A##, I##)
# Empirical speedup vs interpreted Cholesky inverse: 221× at n=30.
#
# Solve on triangular matrices works as forward/back substitution
# (Praat detects triangular structure):
#     # L is lower-triangular Cholesky factor
#     z# = solve# (L##, y#)              # forward solve
#     x# = solve# (transpose## (L##), z#) # back solve
# Empirical speedup vs interpreted substitution: 47× at n=30.
#
# Specialized solvers (all native, all C-speed):
#     solveNonnegative# (A##, y#, maxIter, tolerance, infoLevel)
#         — non-negative least squares
#     solveSparse# (A##, y#, maxNonzeros, maxIter, tolerance, infoLevel)
#         — sparse solution; requires more columns than rows
#         (underdetermined system)
#     solveWeaklyConstrained# (A##, y#, alpha, delta)
#         — regularized solution with weight alpha and tolerance delta

# ----------------------------------------------------------------------------
# §4.4  REDUCTION IDIOMS — TRACE, ROW/COLUMN SUMS, HAT DIAGONAL
# ----------------------------------------------------------------------------
#
# trace(A·B) without forming the full product:

# ----------------------------------------------------------------------------
# §4.7  ONE-SHOT vs HOT-LOOP DIAGNOSTIC
# ----------------------------------------------------------------------------
#
# The vectorized speedups documented in §§4.1–4.4 apply to operations
# called once on a large array OR called inside a hot loop. The
# overhead of calling a native function from the interpreter is
# nonzero. Indicative cost: ~5 µs per call at typical sizes.
#
# Heuristics:
#
# 1. ONE-SHOT, large array (n ≥ ~100): use vectorized op every time.
#    The native op runs at C-speed; the call overhead is amortized
#    across the array. Speedups in §§4.1–4.4 are at n=600 and apply
#    directly.
#
# 2. HOT LOOP, large array per iteration (loop body operates on
#    n ≥ ~100 elements): still use vectorized op inside the loop.
#    Per-iteration savings dwarf per-call overhead.
#
# 3. HOT LOOP, scalar work per iteration: interpreter loop is fine.
#    Wrapping each scalar op in a vector function adds call overhead
#    without algorithmic savings. Example: per-frame Pitch query
#    accumulation does not benefit from vectorization.
#
# 4. MIXED: if a loop has both scalar bookkeeping and a per-iteration
#    bulk operation, the bulk operation gets vectorized; the
#    bookkeeping stays in the interpreter.
#
# If unsure, time it. Praat exposes `stopwatch` for empirical
# comparison: set it before the candidate block, read it after,
# compare against the alternative.
#
# Provenance: §§4.1–4.7 derived from EML LMM Phase 3D vectorization
# audit (15 May 2026) plus targeted empirical re-verification in
# Praat 6.4.65 sandbox (17 May 2026, 75 function probes).
